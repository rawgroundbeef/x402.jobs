The current `POST /resources` handler (lines 142-507 of public-api.ts) does, in order:

1. Destructure body fields: `name`, `description`, `resource_url`, `category`, `tags`, `capabilities`, `extra`, `avatar_url`, `network` (default "solana"), `pay_to`.
2. Validate `name` + `resource_url` (400 "Missing required fields" if missing).
3. Validate `network` against `VALID_NETWORKS` (400 "Invalid network" if not in list).
4. Validate `resource_url` parses as URL (400 "Invalid URL" if not).
5. Compute `normalizedUrl = resource_url.replace(/^https?:\/\//, "")`.
6. Call `fetchX402Metadata(resource_url)` (currently uses raw `fetch` — MUST migrate to `safeFetch` in this plan).
7. If null → 400 "Invalid x402 resource".
8. Compute `cachedAvatarUrl` via `cacheImage(avatar_url || x402.avatarUrl, "avatar")` if either exists.
9. `getOrCreateServer(resource_url, userId)` where userId = `req.apiKey!.created_by || "system"`. 500 if null.
10. Compute effective fields (request overrides x402 defaults): `effectiveNetwork`, `effectivePayTo`, `effectiveDescription`, `effectiveCategory`.
11. Check for existing resource by `normalized_url`. If found:
    a. Determine ownership: `canFullyEdit = isResourceOwner || isServerOwner`.
    b. Build `updates` object: x402 metadata always synced; owner can override user-customizable fields; non-owner only fills missing fields.
    c. If `updates` has keys → UPDATE in DB → return `{ success: true, updated: true, data: {...} }`.
    d. Else → return `{ success: true, updated: false, message: "Resource already exists with all provided fields", data: {...} }`.
12. If no existing resource: generate unique slug via `getUniqueResourceSlug(serverId, baseSlug)`, INSERT, return `{ success: true, created: true, data: {...} }` with HTTP 201.

The shared helper `registerOneResource(input, apiKeyUser)` must return a normalized result object so both endpoints can consume it:

```typescript
type RegisterInput = {
  name?: unknown;
  description?: unknown;
  resource_url?: unknown;
  category?: unknown;
  tags?: unknown;
  capabilities?: unknown;
  extra?: unknown;
  avatar_url?: unknown;
  network?: unknown;
  pay_to?: unknown;
};

type ApiKeyUser = {
  id: string;          // req.apiKey!.id
  name: string;        // req.apiKey!.name
  created_by: string | null; // req.apiKey!.created_by
};

type RegisterResult =
  | {
      status: "created";
      resource: ResourceResponseData; // { id, name, slug, url, resource_url, network, category, server_id, server_slug, created_at }
    }
  | {
      status: "updated";
      resource: ResourceResponseData;
      updated_fields: string[]; // for logging — keys of the `updates` object that were applied
    }
  | {
      status: "noop";
      resource: ResourceResponseData; // existing resource returned as-is (the "updated: false" case)
    }
  | {
      status: "skipped";
      error: "not_owner";
      message: string;
      existing_id?: string; // optional, for caller debugging
    }
  | {
      status: "error";
      error: string;   // short code, e.g. "Invalid x402 resource", "URL not allowed", "Invalid URL", "Invalid network", "Missing required fields"
      message: string; // human-readable details
      http_hint?: 400 | 500; // hint for the single-endpoint handler to map back to a status
    };
```

**Key callout — `skipped` vs `noop`:** the CURRENT single endpoint allows non-owners to fill in missing fields and returns `{ updated: true|false, ... }`. The PRD adds a new "skipped" status to BULK only, defined as "matched by normalized URL BUT caller does NOT own it." For backwards compatibility of the single endpoint, do NOT change the single-endpoint behavior — the non-owner partial-fill path stays. `status: "skipped"` is a BULK-only concept; in the helper, return `skipped` ONLY when a non-owner is matched AND no fields would be updated (i.e., the bulk caller asked to register something someone else already owns and nothing the caller can change is missing). The single-endpoint handler maps `skipped` back to its existing "Resource already exists" / "updated: false" response (preserves contract). Bulk surfaces it as `status: "skipped"` with `error: "not_owner"`.

Single-endpoint mapping from RegisterResult → existing HTTP response shape (do not break this):
- `created` → 201 `{ success: true, created: true, data: result.resource }`
- `updated` → 200 `{ success: true, updated: true, data: result.resource }`
- `noop` → 200 `{ success: true, updated: false, message: "Resource already exists with all provided fields", data: result.resource }`
- `skipped` → 200 `{ success: true, updated: false, message: "Resource already exists with all provided fields", data: result.resource }` (the single endpoint already had this behavior — match it)
- `error` with `http_hint: 400` → 400 `{ error: result.error, message: result.message }`
- `error` with `http_hint: 500` → 500 `{ error: result.error, message: result.message }`

Bulk-endpoint mapping from RegisterResult → per-item result entry (each is wrapped with `index`):
- `created` → `{ index, status: "created", resource: {...} }`
- `updated` → `{ index, status: "updated", resource: {...} }`
- `noop` → `{ index, status: "updated", resource: {...} }` — bulk treats noop as updated=true semantically so callers see "we matched and processed this one" (per PRD; revisit if user disagrees)
- `skipped` → `{ index, status: "skipped", error: "not_owner", message: result.message }`
- `error` → `{ index, status: "error", error: result.error, message: result.message }`

Summary computation: tally each status from the per-item results array. Note `noop`-mapped-to-`updated` increments the `updated` counter.

Concurrency wrapper signature with p-limit:
```typescript
import pLimit from "p-limit";
const limit = pLimit(5);
const results = await Promise.all(
  resources.map((input, index) =>
    limit(async () => {
      try {
        const result = await registerOneResource(input, req.apiKey!);
        return mapResultToBulkEntry(result, index);
      } catch (err) {
        // Defense in depth — registerOneResource should not throw for input/validation errors,
        // but ANY uncaught throw must NOT fail the whole batch.
        console.error(`[bulk] item ${index} threw:`, err);
        return {
          index,
          status: "error" as const,
          error: "Internal error",
          message: err instanceof Error ? err.message : "Unknown error",
        };
      }
    }),
  ),
);
```

Bulk rate limiter (new export in src/middleware/rateLimit.ts):
```typescript
export const bulkResourceRateLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 6,
  standardHeaders: true,
  legacyHeaders: true,
  keyGenerator: (req) =>
    req.apiKey?.id ? `bulk:apikey:${req.apiKey.id}` : `bulk:ip:${req.ip || "unknown"}`,
  handler: (req, res) => {
    res.status(429).json({
      error: "Too Many Requests",
      message: "Bulk rate limit exceeded: 6 requests per minute. Please slow down.",
      retryAfter: res.getHeader("Retry-After"),
    });
  },
  message: {
    error: "Too Many Requests",
    message: "Bulk rate limit exceeded: 6 requests per minute.",
  },
});
```

**Critical:** the `bulk:` keyspace prefix is what makes this bucket orthogonal to `apikey:` used by the existing minuteRateLimiter. Both buckets count separately. Mount order matters: in public-api.ts the existing `router.use(minuteRateLimiter); router.use(hourlyRateLimiter);` applies to ALL routes including bulk. Apply `bulkResourceRateLimiter` as an inline per-route middleware on the bulk handler ONLY, so bulk requests count against both the per-route bulk bucket AND the broader per-minute/hour buckets (consistent with how the user expects bulk traffic to also count against overall usage caps).

Bulk endpoint mount sketch:
```typescript
publicApiRouter.post("/resources/bulk", bulkResourceRateLimiter, async (req, res) => {
  // ... validate body, run pLimit(5) over registerOneResource, aggregate
});
```
