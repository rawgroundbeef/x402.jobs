**Why each script exists:**

| Script | Invokes | Purpose | Source |
|--------|---------|---------|--------|
| `dev:web` | `pnpm --filter @x402jobs/web dev` (= `next dev -p 3010`) | Run web only; useful when api is hosted elsewhere (staging) | PATTERNS section 10 + RESEARCH Pattern 3 line 408-411 |
| `dev:api` | `pnpm --filter x402-jobs-api dev:api` (= nodemon → api on 3011) | Run api only; useful for backend-focused work without Inngest noise | PATTERNS section 10 |
| `dev:inngest` | `pnpm --filter x402-jobs-api dev:inngest` (= `npx inngest-cli@latest dev`) | Run Inngest dev UI only; useful when iterating on workflow definitions | PATTERNS section 10 |
| `test` | `turbo run test` | Run test task across workspaces (today: api only; web TBD) | PATTERNS section 10 |

**Important: `dev:api` invokes api's `dev:api` script, NOT api's top-level `dev` script.**

The api repo's `package.json#scripts` (imported in Plan 02) contains:
```jsonc
{
  "scripts": {
    "dev": "concurrently --kill-others \"pnpm dev:api\" \"pnpm dev:inngest\"",  // unified within api
    "dev:api": "nodemon ...",      // api server only
    "dev:inngest": "npx inngest-cli@latest dev",
    ...
  }
}
```

The root `pnpm dev` → `turbo run dev` invokes the api's `dev` script (which uses `concurrently` to spawn `dev:api` + `dev:inngest`). The fallback `pnpm dev:api` at root SKIPS the concurrently layer and runs api only. Same for `pnpm dev:inngest`. This gives the developer three orthogonal entry points without interfering with the unified path (per RESEARCH Pattern 3 line 414: "The api repo's `dev` script uses `concurrently --kill-others` which means Ctrl-C kills both api and Inngest. Good behavior; preserve.")

| `pnpm.onlyBuiltDependencies` (8-entry union) | Plan 02 Task 6 landed this. |
| `workspaces`, `engines`, `devDependencies` blocks | No need to change. |

**No `engines.node` bump in this plan** despite PATTERNS section 10's note ("may optionally bump from `>=18` to `>=22` to match the api Dockerfile target"). That's a Discretion area in PATTERNS; we leave at `>=18` to maintain compatibility with anyone running Node 18 locally. A future plan can bump if needed.
