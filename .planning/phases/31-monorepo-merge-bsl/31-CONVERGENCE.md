## What Could Have Been Cleaner

- **The repo was already PUBLIC.** The Phase 31 plan was structured around "do all scrubs BEFORE flipping to public" — but the repo never needed flipping. The pre-scrub state on the `plan/31-monorepo-merge-bsl` branch (squash-import commit `4fe9e27`) was publicly visible for ~15 hours between when the branch was pushed (2026-05-15 ~16:36) and when the major scrubs landed (2026-05-16 ~07:24). No live cryptographic secrets, API keys, signing keys, or wallet keys were ever in the repo at any point — verified by full-history grep. What WAS exposed: admin email, two Supabase project IDs (RLS-protected), a third-party Gmail used as a script default, the personal scripts, and one macOS-username path. Treated as footprint-info exposure rather than a credential breach; precautionary `SUPABASE_SERVICE_ROLE_KEY` rotation recommended (low cost, high peace-of-mind).
- **Plan 03 + 05 deployments took multiple iterations.** The new CI workflow surfaced pre-existing lint/typecheck issues (deferred via todos), and the Vercel + Railway monorepo configs took 3 and 3 iterations respectively to converge. Once the Dockerfile / vercel.json / railway.json patterns are settled, future PRs should deploy cleanly.
