**Exact content to write — verbatim, no deviations:**

```ini
# Supply-chain hardening — Phase 30 (2026-05-14)
#
# Reject npm package versions less than 72 hours old at resolution time.
# Most malicious npm publishes are detected and removed within 24-48 hours,
# so a 72-hour delay neutralizes the bulk of supply-chain zero-days.
# Source: https://pnpm.io/blog/releases/10.16
#
# 4320 minutes = 72 hours = 3 days
minimum-release-age=4320

# Internal workspace packages bypass the delay so we can publish urgent
# fixes to @x402jobs/sdk and consume them immediately. Scope-pattern syntax
# documented at https://pnpm.io/settings#minimum-release-age-exclude
minimum-release-age-exclude=@x402jobs/*

# NOTE: frozen-lockfile is intentionally OMITTED here.
# - pnpm enables it automatically when CI=true (Vercel + Railway both set it).
# - Adding it would block local `pnpm add` / `pnpm update` workflows
#   and would have blocked the pnpm 9->10 lockfile regeneration in plan 30-01.
# - See RESEARCH Pattern 3 + Pitfall 4 for the full rationale.
```

**Why this differs from ROADMAP Phase 30 line 451 ("frozen-lockfile=true (CI safety)"):**
RESEARCH explicitly overrides the roadmap on this point — pnpm auto-enables frozen-lockfile when `CI=true`, and putting it in `.npmrc` is actively harmful for local dev and for the pnpm 9-to-10 lockfile regen step we just completed in plan 30-01. The prompt explicitly states: "where the research contradicts ROADMAP.md (... frozen-lockfile ...), the research wins."

**Loading semantics (from RESEARCH Pattern 2, Open Question A1):**
- pnpm reads `.npmrc` from repo root (plus `~/.npmrc`, plus per-package, in that order; repo root is the canonical location for this project)
- `minimum-release-age=4320` applies at RESOLUTION time
- `pnpm install --frozen-lockfile` does NOT re-resolve, so already-locked transitive deps install unimpeded (assumption tested empirically in Task 2 below)
- Without `--frozen-lockfile`, pnpm will re-resolve and CAN refuse to satisfy a `package.json` change that targets a <72h package

**Internal scope verification (from PATTERNS section "@x402jobs/* scope usage"):**
- `apps/web/package.json` is `@x402jobs/web` (consumes `@x402jobs/ui` via `workspace:*`)
- `packages/sdk/package.json` is `@x402jobs/sdk` (published to npm)
- `packages/ui/package.json` is `@x402jobs/ui` (private workspace dep)
- The exclude line is NOT just future-proofing — it actively prevents the gate from blocking workspace resolution of fresh internal versions

**Cross-repo scope:** This `.npmrc` lives in THIS repo only. The sibling api repo does NOT get a `.npmrc` in Phase 30 (the api repo doesn't consume `@x402jobs/*` packages today and a duplicate file would create two policy sources of truth). Phase 31 (Monorepo Merge) folds the api into this tree and a single `.npmrc` covers the merged repo.
