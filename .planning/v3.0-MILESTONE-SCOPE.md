# v3.0 Milestone Scope: Open Source + Agent-Native x402.jobs

## Executive Summary

v3.0 transforms x402.jobs from a closed-source product with a UI-first builder into a source-available, agent-native platform. The backend (currently in a separate private repo) merges into the existing open frontend monorepo under BSL 1.1. A focused internal security pass hardens the codebase before going public. After the merge, we ship an agent SDK + `skill.md` so any LLM with HTTP fetch (Claude, GPT, Gemini, local agents) can discover x402.jobs workflows, build new ones, and pay per step using its own wallet.

The headline external narrative: **"agents bring their own wallet, build workflows, and earn USDC."** The headline internal narrative: **stop pretending we have third-party-grade custody when we don't; eliminate it entirely.**

## Why Now

Three forcing functions converged this week:

1. **Wallet custody audit (today, 2026-05-12)** surfaced that the legacy `encrypted_private_key` column was holding base64 plaintext despite its name. Fixed in Phase 27 with AES-256-GCM at rest, but the broader insight remains: we shouldn't be custodying user wallet keys at all if we don't have to. Going non-custodial for agents (their wallet, not ours) sidesteps the entire class of issues.
2. **Open source pressure.** The x402 ecosystem rewards open infrastructure plays. Competitors and adjacent projects are publishing source. Staying closed signals "SaaS lock-in" in a community oriented around protocols. We get more credibility going BSL than staying private — and BSL keeps the "no one can clone-and-host us commercially" clause intact for 4 years.
3. **The agent-native opportunity.** The agent SDK + `skill.md` lets x402.jobs become the canvas-and-runtime for AI agents that need to build paid workflows. The publish side already exists (any workflow becomes an x402 URL anyone can call) — we just need to make the create + execute surface agent-callable.

## In Scope

- ✅ Wallet keys encrypted at rest (AES-256-GCM)
- Internal security review with tooling (`pnpm audit`, Semgrep, agent-driven structured pass) → `REVIEW.md` with Critical/High/Medium/Low classification
- Fix all Critical + High findings; defer Medium+Low to GitHub issues
- pnpm 9 → 10 upgrade with supply-chain `.npmrc` (`minimum-release-age=4320`, blocked lifecycle scripts)
- Repository merge: import `x402jobs-api` into `x402jobs/apps/api/` via squashed commit
- License declaration: BSL 1.1 with 4-year change date → Apache-2.0
- `SECURITY.md` with bug bounty policy and responsible disclosure
- Public announcement post (open source + agent SDK preview)
- Agent SDK package (`@x402jobs/agent` or similar — naming TBD)
- `/skill.md` endpoint at `x402.jobs/skill.md` (Anthropic Skill format)
- Agent endpoints: `POST /plan`, `POST /runs`, SSE stream for run progress
- Documentation: README rewrite, agent SDK docs, self-host guide
- Verification: a real LLM (Claude with the skill loaded) builds and runs an x402 workflow end-to-end with its own wallet

## Out of Scope (Deferred to v3.1+)

- **Full human-flow convergence to non-custodial (Q1 option A).** Humans stay on the current custodial flow (now with encryption at rest) for v3.0. Migrating humans to passkey smart wallets is the natural v3.1.
- **Agent-native discovery API.** Agents calling each other's published workflows works today via URLs, but there's no agent-shaped discovery (the Bazaar is UI-only). Surface in v3.1.
- **Workflow composition** — agents importing other agents' workflows as primitives.
- **Streaming / dynamic pricing** (Q2 option "support both") — per-token / per-byte priced nodes via SSE relay-and-sign. Deferred unless an LLM-resource use case forces it earlier.
- **Reputation / quality signals** for agents and workflows.
- **Third-party security audit** (Trail of Bits / Spearbit). Cost-prohibitive pre-revenue. Internal pass + community bug bounty is the v3.0 posture.
- **Migration to Turnkey / Privy / similar custody KMS** for the residual human flow. Lateral move; non-custodial is the destination, not "better custody."

### Phase 28: Structured Security Review

**Goal:** Catch additional security issues before going public. Internal pass (Tier 3 of the cost-tier framework) — no third-party engagement.

**Scope:**
- **Tooling layer** (mechanical, ~15 min):
  - `pnpm audit` for known dep CVEs
  - Semgrep with `p/owasp-top-ten` and `p/typescript` rulesets
  - Custom grep patterns: `service_role` usage justification, missing `authMiddleware`, hardcoded user ID comparisons
- **Agent layer** (~1-2h):
  - Spawn security-review agent against high-risk surfaces:
    1. Payment-signing paths (`execute.ts`, `inngest/utils/execute-x402.ts`, `lib/usdc-transfer.ts`, escrow + platform fee utils)
    2. Auth + RLS (`middleware/auth.ts`, all `authMiddleware`-mounted routes, service_role bypass justifications)
    3. Webhook signature verification (Helius, Inngest, external callbacks)
    4. IDOR / cross-user data leaks (every route's user-scoping)
    5. Money math (integer/decimal handling around USDC, fees, escrow)
    6. Replay attacks (can the same x402 payment header be used twice? can the same run be triggered concurrently and double-charged?)
    7. Secret leaks in logs and response bodies
- Output: `REVIEW.md` with severity-classified findings (Critical / High / Medium / Low / Informational)

**Success criteria:**
- Tooling pass complete; findings reviewed
- Agent pass complete; `REVIEW.md` produced
- All Critical and High findings either fixed or explicitly accepted with rationale
- Medium and Low findings filed as GitHub issues with target milestones

**Honest limitations to document:** This is an internal AI-assisted review. We catch OWASP-class issues, common auth/key handling bugs, and what Semgrep flags. We do NOT match a domain-expert human reviewer on protocol-level attacks (subtle EIP-3009 nonce reuse, Solana PDA hijacking, MEV-style griefing). `SECURITY.md` will say this explicitly.

### Phase 29: Bulk Resource Registration (side quest, added 2026-05-13)

**Goal:** Ship `POST /api/v1/resources/bulk` so marketplaces can register up to 25 x402 resources in one API call. Closes the leak in the `/developers` acquisition funnel where the marketing page implies bulk is supported but the API only takes single items.

**Why now (between Critical fixes and the rest of the security work):** marketplaces asking "can I register 100 endpoints in one call?" is a visible product gap today; the Phase 28 Highs are real but invisible to customers. Slot the visible win first, then head back to Highs.

**Scope:**
- `x402jobs-api`: refactor `POST /resources` per-item logic into shared `registerOneResource(input, apiKeyUser)`; add `POST /api/v1/resources/bulk` handler (cap 25, concurrency 5, separate rate-limit bucket 6 req/min)
- Per-item statuses: `created` | `updated` | `skipped` (ownership conflict, reason `not_owner`) | `error`
- Inherit SSRF protection via existing `fetchX402Metadata` → `safeFetch` chain (post-CRIT-07)
- `x402jobs`: `/docs/resources` adds "Bulk registration" section under `#programmatic-registration`
- `x402jobs`: `/developers` "Running a marketplace?" section copy/snippet tweak
- `x402jobs`: homepage `DiscoverSection` adds a bulk-affordance line under the existing tagline
- Test coverage: single-endpoint suite passes unchanged after refactor; new bulk integration tests for partial-success and SSRF inheritance

**Success criteria:**
- `POST /api/v1/resources/bulk` accepts up to 25 items, returns per-item results with HTTP 200 on partial failure
- Existing `POST /resources` tests still pass after refactor (modulo the pre-existing `resource-registration*` exclusion)
- Docs page renders a working code example; `/developers` and homepage copy reflect the bulk capability
- Two PRs merged: one in `x402-jobs-api`, one in `x402jobs`

**Spec:** `.planning/PRD-bulk-resource-registration.md` (approved 2026-05-13)

**Estimated effort:** ~1.5 days

### Phase 30: Supply Chain Hardening

**Goal:** Reduce supply-chain attack surface before going public, since open repos are higher-value targets.

**Scope:**

- First install will surface lifecycle-script approvals; vet each and add to `pnpm.onlyBuiltDependencies` allow list (likely just `sharp`, `esbuild`, `better-sqlite3` if any)
- Drop root `.npmrc`:

  - `minimum-release-age-exclude=@x402jobs/*` (internal packages)
  - `frozen-lockfile=true` (CI safety)
- Verify Railway Dockerfile builds with new pnpm version
- Document the policy in `SECURITY.md`

**Success criteria:**
- Both apps build cleanly under pnpm 10
- `.npmrc` policy applied
- CI green

### Phase 32: Agent SDK Design + skill.md

**Goal:** Concrete design contract before writing any SDK code. Resolves the open decisions from this morning.

**Open decisions to lock down:**

- **Q1: Scope.** Full convergence (passkey smart wallets for humans too) vs agent-first (humans stay custodial for v3.0).
  - **Tentative answer:** Agent-first (B). Human migration deferred to v3.1.
- **Q2: Pricing model.** Mandate static per-node pricing for simplicity, or support both static and dynamic-via-SSE from day one.
  - **Tentative answer:** Static for v3.0. Dynamic-via-SSE deferred to v3.1 unless an LLM-resource use case forces it earlier.
- **Q3: Authorization recipient.** Resource creator direct (invisible facilitator) vs facilitator address we control (better control plane).
  - **No tentative answer.** Need to discuss tradeoffs around refund pooling, telemetry, and trust posture.

**Deliverables:**
- `AI-SPEC.md` for the SDK design (use `gsd-ai-integration-phase` skill — produces a structured contract)
- `skill.md` content: Anthropic Skill format, hosted at `x402.jobs/skill.md`
- Endpoint shape specs for `POST /plan`, `POST /runs`, SSE event stream
- Pre-signed authorization bundle format (EIP-3009 for EVM, partialSign for Solana)
- Reference dataset for evals (agent-built workflows we expect to support)
- Failure mode taxonomy (what can go wrong: pricing drift, expired auths, partial execution, etc.)

**Success criteria:**
- A reader can implement the SDK from the spec without further questions
- All three open decisions resolved with rationale recorded
- Eval plan in place (the design includes how we'll know it works)

### Phase 33: Agent SDK Implementation

**Goal:** Ship the SDK + endpoints; validate with a real LLM building and running a workflow.

**Scope:**
- SDK package: TypeScript first, published to npm under `@x402jobs/agent`

- `/skill.md` served from `apps/web` (or `apps/api` — TBD per design)

- Docs: usage examples, minimal "hello x402.jobs" agent
- E2E validation: Claude with the skill loaded builds a workflow, runs it on testnet (or mainnet with $1 of float), workflow executes and pays

**Success criteria:**
- An external developer can `npm install @x402jobs/agent`, set up a wallet, and run a workflow within 10 minutes of reading the docs
- An LLM with the `/skill.md` fetched (no SDK install) can do the same
- Live demo: Claude builds + runs a multi-step workflow end-to-end

## Notes for Future Maintainers

- The "agents earn $$" framing in the public messaging refers to publishing workflows as x402 URLs — that capability already exists in v2.x and is implicitly unlocked by the agent SDK, not net-new code.
- The wallet encryption work was executed ad-hoc on 2026-05-12 (this conversation) without going through the full GSD phase process. Phase 27's record is retroactive — see commit messages on `rawgroundbeef/x402-jobs-api` for the actual implementation history.
- `WALLET_ENCRYPTION_SECRET` must survive any infrastructure migration. If it's lost, every wallet becomes unrecoverable (the legacy plaintext backup gets deleted after the confidence window). Stash in password manager + treat as critical secret.
